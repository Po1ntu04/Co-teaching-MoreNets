# 实验结果深度分析报告

## 一、实验现象总结

### 1.1 三组实验最终结果对比

| 实验 | 配置 | 最终Test Acc | Peak Test Acc | 最终Train Acc | q_mean | overlap | π_t |
|------|------|-------------|---------------|---------------|--------|---------|-----|
| Exp1 | γ=0.5, replay=0 | ~47-48% | ~75.9% (E19) | 99.98% | 0.999 | 0.569 | 0.991 |
| Exp2 | γ=0.5, explore=0.1 | ~48-49% | ~76.2% (E16) | 99.98% | 0.999 | 0.570 | 0.991 |
| Exp3 | Full System (replay=5000) | ~48-49% | ~77.2% (E13) | 99.99% | 0.999 | 0.742 | 0.993 |

### 1.2 关键时间节点分析

观察训练曲线，可以识别三个明显阶段：

**阶段1：健康学习期 (Epoch 1-15)**

- Test Accuracy 稳步上升：10% → 75%+
- Train Accuracy 缓慢上升：20% → 46%
- q_mean 从 0.78 下降至 0.55（模型在区分干净/噪声样本）
- overlap 从 0.99 下降至 0.97（模型开始分化）

**阶段2：转折点 (Epoch 16-25)**

- Test Accuracy 开始下降
- Train Accuracy 继续上升
- q_mean 开始上升（预示问题）

**阶段3：过拟合/认知固化期 (Epoch 25+)**

- Train Accuracy → 99.9%+
- Test Accuracy → 47-49%（仅比随机猜测好）
- q_mean → 0.999（所有样本都被认为是"干净"的）
- π_t → 0.99（先验完全偏离真实值 0.5）

## 二、问题根因深度分析

### 2.1 核心问题一：Beta+EMA 双保险为何失效？

**target.md 中的设计意图**：
> π_t ← EMA(batch-mean(q))，使用 Beta 先验正则化

**实际代码实现**（main.py 485-490行）：

```python
sum_q = float(q_batch.sum().item())
a = args.pi_beta_a  # 默认=2.0
b = args.pi_beta_b  # 默认=2.0
pi_hat = (a + sum_q) / (a + b + float(batch_size))  # batch_size=256
pi_t = args.pi_ema * pi_t + (1.0 - args.pi_ema) * pi_hat  # pi_ema=0.99
```

**失效原因分析**：

1. **Beta 先验太弱**：a=b=2 相对于 batch_size=256 几乎没有正则化效果
   - 当 sum_q ≈ 230（q_mean=0.9），pi_hat = (2+230)/(4+256) ≈ 0.89
   - 先验贡献：4/(4+256) = 1.5%，几乎可忽略

2. **EMA 太慢但方向错误**：pi_ema=0.99 确实让 π_t 变化很慢
   - 但问题是：q_batch 本身就已经被污染了
   - π_t 会持续、缓慢地向错误方向（1.0）移动
   - "慢变量"只能延缓崩溃，不能阻止

3. **根本问题：q 的计算依赖模型预测 p(y|x)**

   ```python
   q_batch = (π_t * p_ens_y) / (π_t * p_ens_y + (1-π_t)/C)
   ```

   - 当模型记忆噪声标签后，p(y|x) 对所有样本都升高
   - q 自然升高 → π_t 升高 → 形成正反馈
   - **这是一个结构性问题，EMA/Beta 无法从根本上解决**

### 2.2 核心问题二：Clean Replay 是否加剧了认知固化？

**实验对比关键发现**：

| 实验 | replay_size | 最终 overlap | 最终 π_t |
|------|-------------|--------------|----------|
| Exp1 | 0 | 0.569 | 0.991 |
| Exp3 | 5000 | **0.742** | **0.993** |

**Exp3 启用了 replay 后 overlap 反而更高！**

**问题分析**：

1. **Replay 的入选条件存在致命缺陷**：

   ```python
   if qv < args.replay_tau:  # replay_tau=0.8
       continue
   # 只有 q > 0.8 的样本才能进入 replay
   ```

   - 这个判断依赖于 q，而 q 本身会被污染
   - 一旦模型开始记忆，所有样本 q 都会 > 0.8
   - **噪声样本也会进入 replay buffer**

2. **Replay 的使用方式强化了错误记忆**：

   ```python
   # replay 样本直接拼接到当前 batch
   images = torch.cat([images, replay_imgs], dim=0)
   labels = torch.cat([labels, replay_lbls], dim=0)
   ```

   - replay 样本（包含噪声）被重复训练
   - 模型对这些噪声样本的记忆更加巩固
   - **形成"认知固化"：错误知识被反复强化**

3. **没有 replay 样本的"过期淘汰"机制**：
   - replay buffer 满后随机替换
   - 早期（正确识别时）加入的干净样本会被替换
   - 后期（记忆噪声后）加入的噪声样本逐渐占主导

### 2.3 核心问题三：Recent Window 对数据分布的影响

target.md 提到的 "Recent Window (W)" 在当前代码中：

**实现方式**：普通的 epoch 遍历 + shuffle

```python
train_loader = torch.utils.data.DataLoader(
    dataset=train_dataset,
    batch_size=batch_size,
    shuffle=True,  # 每个 epoch shuffle
)
```

**潜在问题**：

- 每个 epoch 都能看到完整数据集
- 没有真正的"窗口"概念
- 模型有无限次机会记忆所有样本

### 2.4 辅助问题：模型同质化

**数据证据**：overlap 从 0.99 最低降到 ~0.57 后趋于稳定

- 所有模型选择相似的样本
- Co-teaching 的"互教纠错"机制失效
- 当 3 个模型都记住同一个噪声样本时，没有任何机制能纠正

## 三、为什么 target.md 的设计会失效：结构性分析

### 3.1 设计假设 vs 实际情况

| target.md 假设 | 实际情况 |
|----------------|----------|
| p(y\|x,z=1) 由干净模型提供 | 模型会逐渐记忆噪声 |
| q 能区分干净/噪声样本 | q 依赖模型预测，会被污染 |
| π_t 作为慢变量能稳定系统 | 慢变量只延缓崩溃，不阻止 |
| Replay 保存高置信度样本 | 高置信度 ≠ 真正干净 |

### 3.2 自证循环的完整链条

```
┌─────────────────────────────────────────────────────────────────────┐
│                        完整自证循环链条                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  模型记忆噪声 ──► p(y|x) 对噪声样本升高 ──► q 升高 ──┐              │
│       ▲                                                │              │
│       │                                                ▼              │
│       │                                    π_t = EMA(mean(q)) 升高   │
│       │                                                │              │
│       │                                                ▼              │
│       │                              噪声样本获得高训练权重          │
│       │                                                │              │
│       │                                                ▼              │
│       │                              噪声样本进入 Replay Buffer      │
│       │                                                │              │
│       │                                                ▼              │
│       └──────── 模型被反复训练在这些噪声样本上 ◄───────┘              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.3 关键洞察

**问题不在于 EMA/Beta 参数设置，而在于整个 q 计算框架的结构性缺陷**：

1. **q 的计算使用模型自己的预测**：这本质上是用模型来评估自己，必然导致自证循环

2. **没有外部锚点**：系统缺乏不依赖模型预测的"真实信号"

3. **Replay 放大了问题**：本意是稳定训练，但实际上固化了错误

## 四、设计改进方向（供讨论）

基于以上分析，核心问题是 **q 的计算完全依赖模型预测**，而模型预测会被噪声污染。以下是几个可能的改进方向：

### 4.1 方向一：引入不依赖模型预测的 q 信号

**思路**：q 不应该只看 p(y|x)，还需要"外部锚点"

- **损失排名（Rank-based q）**：用 batch 内相对排名替代绝对损失
  - 优点：排名分布固定，不受模型记忆影响
  - 缺点：失去跨 batch 的一致性
  
- **损失历史对比**：比较当前损失与历史 EMA
  - 如果样本损失"异常下降"，可能是记忆信号

### 4.2 方向二：限制 π_t 的变化范围

**思路**：利用已知噪声率约束 π_t

- 设置 pi_max = 1 - noise_rate + margin
- 使用更强的 Beta 先验（增大 a, b）

**问题**：这是"治标不治本"，可能只是延缓崩溃

### 4.3 方向三：改进 Replay Buffer 的入选和使用机制

**当前问题**：Replay 基于 q 选择，而 q 会被污染

**可能的改进**：

- **早期锁定**：只在训练早期（如前 10 个 epoch）收集 replay 样本，此时 q 估计相对准确
- **使用 loss 排名而非 q**：选择 loss 持续较低的样本
- **定期清洗**：定期用当前模型重新评估 replay 样本，剔除可疑样本
- **时间衰减**：旧的 replay 样本权重逐渐降低

### 4.4 方向四：模型多样性强制

**思路**：防止模型趋同，保持 Co-teaching 的纠错能力

- **强化 explore 机制**：更早触发、更大比例
- **差异化初始化**：使用不同的随机种子
- **对抗性训练**：显式鼓励模型对边界样本给出不同预测

### 4.5 方向五：重新审视"流式"假设

**思路**：当前实现并非真正的流式学习

- 每个 epoch 都能看到完整数据集
- 模型有无限次机会记忆

**可能的改进**：

- 限制每个样本的最大训练次数
- 真正的单遍流式：每个样本只过一次

## 五、待讨论的问题

1. **Beta+EMA 参数选择**：当前 a=b=2, pi_ema=0.99 是否合理？应该如何调整？

2. **Replay 的角色**：
   - target.md 中 Replay 的设计意图是"稳定训练"
   - 但实验显示它可能加剧了认知固化
   - 是否应该禁用 Replay？或者彻底重新设计入选机制？

3. **q 的计算基础**：
   - Posterior-q 依赖 p(y|x)，而这正是会被污染的
   - 是否有更鲁棒的 q 计算方式？

4. **早停 vs 长期稳定**：
   - 早停可以避免过拟合，但这是"逃避问题"
   - 理想情况下，系统应该能自我稳定
